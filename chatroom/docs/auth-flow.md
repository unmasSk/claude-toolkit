# Authentication Flow

## Overview

Authentication uses short-lived, one-time-use UUID tokens. There are no sessions, cookies, or JWTs. Every WebSocket connection consumes exactly one token.

## Step-by-Step

### 1. Obtain a token

```
POST /api/auth/token
Content-Type: application/json

{ "name": "alice" }
```

Response:
```json
{
  "token": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "expiresAt": "2026-03-19T14:30:00.000Z",
  "name": "alice"
}
```

**Name rules:**
- 1–32 characters, `[a-zA-Z0-9_-]` only
- Cannot be empty or missing
- Cannot be a reserved agent name (`bilbo`, `cerberus`, `argus`, etc.)
- Cannot be `claude` (orchestrator identity) or `system`
- `user` is allowed — it is the default frontend identity

**Rate limit:** 20 requests per 60 seconds (global, not per-IP).

**Token properties:**
- UUID v4, generated by `crypto.randomUUID()`
- TTL: 30 minutes
- Store cap: 10,000 tokens max

### 2. Connect via WebSocket

```
WS /ws/default?token=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx&name=alice
```

The `name` query parameter is **ignored** for the author identity — the name bound to the token at issuance is used. The `?name=` param exists for display purposes only.

On `open()`, the server:
1. Checks Origin header against `WS_ALLOWED_ORIGINS`
2. Checks WS upgrade rate limit (50/second global)
3. Checks per-room connection cap (20 max)
4. Calls `validateToken(token)` — validates and **deletes** the token (one-time-use)
5. Assigns a `connId`, tracks state in `connStates` and `roomConns`
6. Subscribes connection to `room:{roomId}` pub/sub topic
7. Sends `user_list_update` broadcast
8. Sends `room_state` with recent messages and agent statuses

**Token errors:** If the token is missing, unknown, or expired, the server sends an `error` message and closes the connection.

### 3. Invite route authentication (non-consuming)

```
POST /api/rooms/default/invite
Authorization: Bearer xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
Content-Type: application/json

{ "agents": ["bilbo", "cerberus"] }
```

The invite endpoint uses `peekToken()` (not `validateToken()`). The token is **not consumed** — the caller can still use it for their WS connection.

Rate limit: 20 requests per 60 seconds (separate bucket from `/auth/token`).

## Token Lifecycle

```
issueToken("alice")
  → tokens.set(uuid, { name: "alice", expiresAt, tokenBuf })
  → return { token: uuid, expiresAt }

peekToken(uuid)  [for /invite]
  → validate TTL + constant-time compare
  → return "alice"  (token stays in map)

validateToken(uuid)  [for WS open]
  → validate TTL + constant-time compare
  → tokens.delete(uuid)     ← ONE-TIME USE
  → return "alice"

GC timer (every 10 min)
  → remove all entries where Date.now() > expiresAt
```

## Constant-Time Comparison

Token comparison uses `crypto.timingSafeEqual()` to prevent timing-based token enumeration. Map.get() locates the entry by key (a necessary pre-check), then `timingSafeEqual` compares the full token in constant time. Buffer lengths are checked before calling `timingSafeEqual` because it throws on length mismatch.

## Brute-Force Detection

Auth failures are tracked by the first 8 characters of the presented token (the "source prefix"). After 10 failures within 60 seconds from the same prefix, a `logger.error` fires at level ERROR. The failure map is capped at 5,000 entries. Stale windows are GC'd every 10 minutes alongside token expiry.

## Author Identity Enforcement

The author name on messages is **always set server-side**. The client cannot inject an author name via the WS message body. In `send_message` and `invoke_agent` handlers:

```ts
const connState = connStates.get(connId);
const authorName = connState.name; // from token, never from message body
```

This prevents impersonation even if a malicious client sends `{ type: 'send_message', author: 'cerberus', content: '...' }`.
